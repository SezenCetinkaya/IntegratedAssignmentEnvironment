Integrated Assignment Environment (IAE) — User Manual

CE316 Project — Izmir University of Economics. IAE is an offline desktop tool for batch-evaluating student programming submissions. It unzips student archives, compiles or interprets their code, compares output to an expected result, and presents a results table that can be exported as CSV.

1. Introduction

IAE automates the repetitive work of grading programming assignments:

  1. Each student submits a ZIP file containing their source code.
  2. IAE extracts every ZIP into a temporary workspace.
  3. For compiled languages (C, Java) IAE compiles the source. For interpreted languages (Python) it skips this step.
  4. IAE runs the program and captures its standard output.
  5. The output is compared line-by-line against an expected output file you provide.
  6. Results (PASS / FAIL / compile error / runtime error) are shown in a table and can be saved to the database or exported as CSV.

A Configuration holds the language-specific settings (compiler path, run command, timeout). A Project links one configuration to a submission folder and an expected output file. You can have many projects (one per assignment) each reusing the same configuration.

Click Save when done. The configuration is stored in the local SQLite database.

3. Creating a New Project

Each assignment you want to grade is a Project. Go to File → New Project.

  1. Project Name — give it a descriptive name, e.g. HW1 - Sorting Algorithms.
  2. Configuration — select the language configuration you created in Section 2 from the drop-down.
  3. Submission Directory — click Browse and select the folder that contains all student ZIP files.
  4. Expected Output File — click Browse and select the .txt file that contains the correct program output (see Section 5).
  5. Run Arguments — optional; see Section 6.
  6. Click Save Project.

Once saved, click Run Evaluation to start grading. IAE processes all ZIPs in the submission directory. A progress indicator appears while running. If a student's submission fails, IAE continues with the next one — one broken ZIP does not stop the batch.

4. Preparing the ZIP Submission Directory

All student ZIP files must be placed in a single flat folder. IAE uses the ZIP filename (without extension) as the student identifier.

Required folder structure

submissions/
    20210001.zip
    20210002.zip
    20210003.zip
    ...

Required ZIP contents

Each ZIP must contain the source file at its root or in a single subdirectory. IAE searches for the filename defined in the configuration's Source Filename field (case-insensitive).

20210001.zip
    main.c          ← IAE finds this (Source Filename = "main.c")

or

20210001.zip
    20210001/
        main.c      ← also accepted (one level of subdirectory)
Do not nest ZIPs inside ZIPs. IAE only extracts one level.
Student ID as filename: Name each ZIP after the student's ID (e.g. 20210001.zip). This ID appears in the results table and CSV export.

5. Expected Output File Format

Create a plain text file containing exactly what the correct program should print to standard output. IAE compares each student's output to this file line by line.

Rules

  • Save the file as UTF-8.
  • Line endings: IAE normalises both CRLF (Windows) and LF (Unix) automatically, so either is acceptable.
  • Trailing blank lines at the end of the file are ignored during comparison.
  • Leading and trailing whitespace on each line is not stripped — match the exact spacing your program produces.

Example

If the assignment asks the program to print the numbers 1 to 5, your expected output file should contain:

1
2
3
4
5
Tip: Run your reference solution once, redirect its output to a file (program.exe > expected.txt), and use that file directly in IAE.

6. Run Arguments

Some programs require command-line arguments when they run (e.g. an input filename or a numeric parameter). Enter these in the Run Arguments field of the project.

When to use

  • The assignment specification says students should accept a filename as argv[1].
  • The program reads a seed or configuration value from the command line.

Format

Enter arguments exactly as you would type them after the program name in a terminal. Multiple arguments are separated by spaces.

ScenarioRun Arguments fieldEffective command
No arguments(empty)main.exe
Input fileinput.txtmain.exe input.txt
Two arguments10 ascendingmain.exe 10 ascending
The same run arguments are passed to every student's program. If the argument is a filename, that file must be present inside every student's ZIP.

7. Viewing Results and Exporting CSV

Results table

After Run Evaluation completes, the Results tab shows one row per student:

ColumnMeaning
Student IDZIP filename without extension
Compile StatusOK, FAILED, or SKIPPED (interpreted languages)
Run StatusPASS, FAIL, ERROR, or TIMEOUT
Execution Time (ms)How long the program ran

Click any row to see the full compile error log or runtime error details in the panel below the table.

Exporting to CSV

  1. Go to File → Export Results... (or use the Export button in the Results tab).
  2. A Save dialog appears. Choose a location and filename, then click Save.
  3. The CSV file can be opened directly in Excel or any spreadsheet application.

CSV column order: studentId, compileStatus, runStatus, executionTimeMs, error

Tip: Open the CSV in Excel, select the executionTimeMs column, and sort descending to quickly spot unusually slow or infinite-loop submissions.

8. Importing and Exporting Configurations

You can share a configuration between machines (e.g. your office PC and a grading laptop) using the export/import feature.

Exporting

  1. Open Configuration → Manage Configurations.
  2. Select the configuration you want to share.
  3. Click Export and choose a save location. IAE saves a .properties file.

Importing

  1. Open Configuration → Manage Configurations on the target machine.
  2. Click Import and select the .properties file.
  3. IAE loads all fields from the file and saves it to the local database.
The Compiler Path is stored verbatim. If the compiler is installed at a different path on the target machine, edit the configuration after importing.

9. Compiled vs Interpreted Languages

IAE handles the two language categories differently:

Compiled (C, Java)Interpreted (Python)
Interpreted checkboxUncheckedChecked
Compile stepRuns Compiler Path + Compile Arguments + Source FilenameSkipped entirely
Run stepRuns Run Command (e.g. main.exe or java Main)Runs Run Command (e.g. python main.py)
Compile Status in resultsOK or FAILEDSKIPPED

C example walkthrough

  1. IAE extracts 20210001.zip to a temporary folder.
  2. It finds main.c in that folder.
  3. Runs: gcc -o main main.c
  4. If compilation succeeds, runs: main.exe (with optional run arguments).
  5. Captures stdout, compares to expected output.

Python example walkthrough

  1. IAE extracts 20210001.zip to a temporary folder.
  2. It finds main.py in that folder.
  3. Compile step: skipped.
  4. Runs: python main.py (with optional run arguments).
  5. Captures stdout, compares to expected output.

10. Troubleshooting

COMPILER_NOT_FOUND

Symptom: Compile Status shows FAILED; error log says COMPILER_NOT_FOUND: gcc.

Cause: The compiler executable is not found at the path specified in the configuration and is not on the system PATH.

Fix:

  • Install the compiler (e.g. MinGW for GCC on Windows, JDK for javac).
  • Either add the compiler's bin/ folder to the system PATH, or enter the full path in the Compiler Path field (e.g. C:\MinGW\bin\gcc.exe).

Source Not Found

Symptom: Run Status shows ERROR; error log says the source file could not be located.

Cause: The student's ZIP does not contain a file matching the Source Filename in the configuration (search is case-insensitive but the name must match).

Fix:

  • Check that the Source Filename in the configuration matches what students were asked to submit (e.g. Main.java vs main.java).
  • Open a few student ZIPs manually to verify the filename they used.

Runtime Error / Non-zero Exit Code

Symptom: Run Status shows ERROR after compilation succeeded.

Cause: The student's program crashed or exited with a non-zero code (e.g. segfault, uncaught exception, missing input file).

Fix:

  • Click the student's row in the results table to see the stderr output for clues.
  • If the program requires an input file via run arguments, verify that file exists inside the student's ZIP.

TIMEOUT

Symptom: Run Status shows TIMEOUT.

Cause: The student's program ran longer than the Timeout value set in the configuration. IAE kills the process automatically.

Fix:

  • If the timeout is too short for a legitimate program, increase Timeout (seconds) in the configuration.
  • A timeout usually means an infinite loop in the student's code — no fix on your side is needed.

Output is FAIL despite seemingly correct output

Symptom: You can see the output looks right, but IAE reports FAIL.

Common causes:

  • Extra or missing trailing newline in the expected output file.
  • Trailing spaces at the end of a line.
  • The program prints to stderr instead of stdout — IAE only compares stdout.
  • Encoding mismatch: save the expected output file as UTF-8.
